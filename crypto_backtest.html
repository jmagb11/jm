Dynamic portfolio backtesting project on crypto currencies

A backtesting test on cryptocurrency returns is the subject of my work below. I tested a strategy based on indicators such as RSI and volatility.

It was a very interesting project, and I asked myself a lot of questions to understand why my code didn’t work in certain cases, or even why my strategy didn’t work. I had to rework certain points several times and even abandon some of them.

I tested simple strategies and ML based strategies.

I justify my choice to test a strategy on the cryptocurrency market, which operates 24 hours a day. When we want to see how the market will behave after the weekend, we look at cryptocurrencies when the stock markets are closed.

Furthermore, this market has no interruptions, compared to other markets where there can be large differences between the opening price and the previous day’s closing price.

I initially wondered why we were doing backtesting in the crypto sector, given that if we do backtesting, in any case, a crypto can have a sharp rise one year and completely fall off the next…

Because backtesting is NOT used to predict which crypto will explode. It is used to answer much deeper questions. Is a systematic rule or an arbitrary choice? Does diversification reduce risk? (in this sector) Is a process better than pure intuition?

My crypto data set contains an RSI_14 column, which calculates the RSI over the last 14 days.

For my strategy, I decided to use the RSI_14. I have a total of €100,000 to invest, and I buy when I am oversold and sell when I am overbought. I have set my oversold zone between 0 and 30 and my overbought zone between 80 and 100. Initially, it was 70-100, but I really want to sell at the extremes and not rush into anything.

I then tried to calculate the average of my RSI overbought and oversold values per day, but I only got averages of 5 or 4. I wondered how it was possible to have only 5 consecutive days of overbought or oversold conditions when, if you look at the charts on Trading View, for example, overbought conditions last much longer on a 14-day RSI. But in fact, the chart is smoothed because it is continuous, which distorts the interpretation somewhat.

An episode of overbought or oversold can last a long time, so the maximum doesn’t tell me much because there may only be one in 50, for example. So I used the median. And I added the total scenario as well to see.

LOW <- 30                                    # I set the oversold RSI threshold.
HIGH <- 80                             # I set the overbought RSI threshold.
RSI_COL <- "RSI_14"                                 # point to the column where the RSI values are stored.

run_lengths <- function(cond) {                     # create a helper to measure consecutive TRUE streaks.
  r <- rle(cond)                              # I compress the logical vector into runs.
  r$lengths[r$values]                    # I return only the lengths of the TRUE runs.
}

tab_rsi <- do.call(rbind, lapply(names(data), function(sym) {     # Then I compute the stats for each crypto and stack everything together.

  tot_overbought <- sum(rsi > HIGH, na.rm = TRUE)               # I count how many days RSI is above HIGH.
  tot_extremes <- sum((rsi < LOW) | (rsi > HIGH), na.rm = TRUE)         # count days in either extreme zone.

    symbol = sym,                                         # I keep the symbol as an identifier.

With regard to volatility, I used a Vol_30 column, which calculates the average volatility over the last 30 days.

WINDOW <- 30                                              # I set the rolling window size (in days) for volatility.

run_lengths <- function(x) rle(x)$lengths[rle(x)$values]     # I use a small helper to get consecutive TRUE streak lengths.

tab_vol <- do.call(rbind, lapply(names(data), function(sym) {                          # Then I compute the same volatility stats for each crypto and bind rows.
  df <- data[[sym]]                                      # grab the dataset for the current symbol.

  ret <- diff(log(df$Adj.Close))                           # I compute log returns from adjusted close prices.

  vol <- sapply(seq_along(ret), function(i)                       # calculate rolling volatility (std dev of returns).
    if (i < WINDOW) NA else sd(ret[(i-WINDOW+1):i], na.rm = TRUE)
  )
  vol <- vol[!is.na(vol)]                                    # I drop the initial NA values where the window is incomplete.

  low <- quantile(vol, 0.30)                            # I set a “low vol” threshold using the 30th percentile.
  high <- quantile(vol, 0.70)                                  # set a “high vol” threshold using the 70th percentile.

  data.frame(
    symbol = sym,                                         # keep the symbol as the identifier.

    vol_mean = mean(vol),                           # I store the average rolling volatility.
    vol_max = max(vol),                                               # store the maximum rolling volatility observed.

    total_days_low_vol = sum(vol < low),                                 # I count how many days fall into low volatility.
    total_days_high_vol = sum(vol > high),                    # I count how many days fall into high volatility.

    mean_days_low_vol = mean(run_lengths(vol < low)),                # compute average length of low-vol streaks.
    median_days_low_vol = median(run_lengths(vol < low)),                # compute median length of low-vol streaks.
    max_days_low_vol = max(run_lengths(vol < low)),                              # I grab the longest low-vol streak.

    mean_days_high_vol = mean(run_lengths(vol > high)),              # I compute average length of high-vol streaks.
    median_days_high_vol = median(run_lengths(vol > high)),             # compute median length of high-vol streaks.
    max_days_high_vol = max(run_lengths(vol > high)),                           # grab the longest high-vol streak.

    stringsAsFactors = FALSE                                   # I prevent strings from being converted into factors.
  )
}))

I then allocate funds based on the inverse volatility of my crypto portfolio. In other words, the more volatile the asset, the lower its weighting in the portfolio.

INIT<-1e5; LOW<-30; HIGH<-80; TCOST<-0.001                           # I set the main parameters (capital, RSI thresholds, costs).

w<-setNames(alloc_table$allocation_pct/100, alloc_table$symbol)
data_bt<-data[names(data)%in%names(w)] ; w<-w[names(data_bt)] ; w<-w/sum(w)
syms<-names(data_bt)
dates<-sort(Reduce(intersect,lapply(data_bt,\(d) as.Date(d$Date))))      # I align all assets on the same common calendar.

P<-S<-matrix(NA_real_,length(dates),length(syms),dimnames=list(as.character(dates),syms))
for(j in seq_along(syms)){ d<-data_bt[[syms[j]]]; d$Date<-as.Date(d$Date); m<-match(dates,d$Date)
  P[,j]<-d$Adj.Close[m] ; S[,j]<-d$RSI_14[m]
}

bt <- function(tr){                                # I run the backtest with tranche size tr.
  tgt<-INIT*w ; step<-tr*tgt ; cash<-INIT
  u<-rep(0,length(syms)) ; eq<-numeric(length(dates)) ; cost<-0

  for(t in seq_along(dates)){
    p<-P[t,] ; r<-S[t,] ; pos<-u*p

    for(j in which(!is.na(r)&r>HIGH&u>0&!is.na(p))){                              # sell on overbought signals.
      amt<-min(step[j],pos[j]) ; su<-amt/p[j] ; c<-amt*TCOST
      u[j]<-u[j]-su ; cash<-cash+(amt-c) ; cost<-cost+c
    }

    pos<-u*p
    for(j in which(!is.na(r)&r<LOW&!is.na(p))){                           # I buy on oversold signals up to target.
      gap<-tgt[j]-pos[j] ; if(gap<=0||cash<=0) next
      amt<-min(step[j],gap,cash) ; c<-amt*TCOST
      u[j]<-u[j]+(amt-c)/p[j] ; cash<-cash-amt ; cost<-cost+c
    }

    eq[t]<-cash+sum(u*p,na.rm=TRUE)
  }

  re<-diff(log(eq))
  yrs<-as.numeric(difftime(tail(dates,1),head(dates,1),units="days"))/365.25
  cagr<-(tail(eq,1)/INIT)^(1/yrs)-1
  dd<-eq/cummax(eq)-1 ; mdd<-min(dd,na.rm=TRUE)
  sh<-if(is.na(sd(re,na.rm=TRUE))||sd(re,na.rm=TRUE)==0) NA else mean(re,na.rm=TRUE)/sd(re,na.rm=TRUE)*sqrt(365.25)
  cal<-if(mdd==0) NA else cagr/abs(mdd)

  list(
    eq=data.frame(Date=dates, Equity=eq),
    stats=c(tranche=tr, final=tail(eq,1), cagr=cagr, sharpe=sh, max_dd=mdd, calmar=cal, trade_cost=cost)
  )
}

I don’t keep a percentage in cash because we can see that It is spent gradually in the back test.

dsr <- function(r, n_trials = 1) {
  r <- r[is.finite(r)]                 # I clean the returns by removing NA and infinite values.
  if (length(r) < 30) return(NA)                  # I require a minimum number of observations.

  sr <- mean(r) / sd(r)                         # compute the standard Sharpe ratio.
  if (!is.finite(sr)) return(NA)                # I stop if the Sharpe is not well-defined.

  s <- skewness(r); k <- kurtosis(r)                     # compute skewness and kurtosis of returns.
  Tn <- length(r)                                 # I store the sample size.

  se <- sqrt((1 - s*sr + ((k-1)/4)*sr^2) / (Tn - 1))               # I estimate the standard error of the Sharpe ratio.
  sr_star <- qnorm(1 - 1/n_trials) * se          # compute the luck-adjusted Sharpe threshold.

  pnorm((sr*(1 + s*sr/6 - (k-3)*sr^2/24) - sr_star) / se)     # I return the Deflated Sharpe Ratio.
}

tree_prob <- function(d){
  if (is.null(d$ret) || length(d$ret) == 0)
    d$ret <- d$Adj.Close / dplyr::lag(d$Adj.Close) - 1     # I compute daily returns if they are not already available.

  r5 <- zoo::rollapply(d$ret, 5, mean, fill = NA, align = "right")  # I compute a 5-day rolling mean of returns.

  X <- data.frame(r1=d$ret, r5=r5, rsi=d$RSI_14, vol=d$Vol_30)           # I build the feature set for the decision tree.
  y <- dplyr::lead(d$ret) > 0                                      # I define the target: next-day positive return or not.

  tr <- na.omit(cbind(y, X))                                       # I remove observations with missing values.
  predict(rpart(y ~ ., tr, method="class"), X, type="prob")[,2]    # I return the probability of a positive return.
}

I therefore explored a machine learning approach to complement the simple strategy. The goal is not to trade directly with the tree, but to better understand the conditions favorable to an increase.

fit_tree <- function(d){
  if (is.null(d$ret) || length(d$ret) == 0)
    d$ret <- d$Adj.Close / dplyr::lag(d$Adj.Close) - 1

  X <- data.frame(
    r1  = d$ret,
    r5  = zoo::rollapply(d$ret, 5, mean, fill=NA, align="right"),
    rsi = d$RSI_14,
    vol = d$Vol_30
  )
  y <- dplyr::lead(d$ret) > 0

  tr <- na.omit(cbind(y, X))
  rpart(y ~ ., data=tr, method="class",
        control=rpart.control(cp=0.01, maxdepth=4),
        model=TRUE)
}

Conclusion :

We compared different strategies, using RSI, whether passive or active, on a set of cryptocurrencies that make up a portfolio. We noticed better performance from active strategies using RSI and volatility. This can be seen with the RSI_tree strategy. There is a very high level of risk, but we accept this risk knowing which market we are operating in. We were able to question the assumption of normality by studying returns and exposing the presence of asymmetry and fat tails. We then coupled the study of our results with a deflated Sharpe ratio because the Sharpe ratio was not sufficient here since we were testing strategies. With the DSRs obtained, we note that the statistical robustness of the various strategies is moderate. Finally, this project shows evidence of a trade-off between return and risk. We noticed that decisions based on systematic rules yield more consistent results than purely arbitrary choices. We also observed that diversification reduces risk, even if it remains limited in the cryptocurrency sector. Finally, backtesting shows that a structured process is generally more reliable than intuition alone, especially in a highly volatile environment.
